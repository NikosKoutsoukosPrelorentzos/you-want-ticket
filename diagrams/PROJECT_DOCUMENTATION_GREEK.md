# Τεκμηρίωση Συστήματος "You Want Ticket"

## 1. Εισαγωγή

### 1.1 Στόχοι της εργασίας
Η παρούσα εργασία αποσκοπεί στον σχεδιασμό και την υλοποίηση μιας σύγχρονης διαδικτυακής εφαρμογής (Web API) για τη διαχείριση και κράτηση εισιτηρίων εκδηλώσεων. Οι κύριοι στόχοι περιλαμβάνουν:
- Την ανάπτυξη ενός εύχρηστου RESTful API με τη χρήση του **FastAPI**.
- Την διασφάλιση της ακεραιότητας των δεδομένων και της ορθής διαχείρισης της διαθεσιμότητας των εισιτηρίων.
- Την εφαρμογή προτύπων ασφαλείας για την αυθεντικοποίηση και εξουσιοδότηση χρηστών (JWT Auth).
- Την εξοικείωση με τη χρήση σχεσιακών βάσεων δεδομένων (**PostgreSQL**) σε περιβάλλον **Docker**.

### 1.2 Ορισμός του προβλήματος προς επίλυση
Το κύριο πρόβλημα που καλείται να λύσει η εφαρμογή είναι η αυτοματοποιημένη διαχείριση των κρατήσεων για εκδηλώσεις, αποφεύγοντας προβλήματα όπως:
- **Υπερκράτηση (Overbooking):** Διασφάλιση ότι δεν θα πωληθούν περισσότερα εισιτήρια από τα διαθέσιμα.
- **Δέσμευση πόρων:** Διαχείριση "ορφανών" κρατήσεων που ξεκινούν αλλά δεν ολοκληρώνονται, μέσω ενός μηχανισμού αυτόματης αποδέσμευσης (Order Cleanup).
- **Ασφάλεια:** Διαχωρισμός ρόλων μεταξύ Διοργανωτών (που δημιουργούν εκδηλώσεις) και Πελατών (που αγοράζουν εισιτήρια).

---

## 2. Σύντομη παρουσίαση της AGILE SCRUM
Η Agile Scrum είναι ένα πλαίσιο διαχείρισης έργων που βασίζεται στην επαναληπτική (iterative) και σταδιακή (incremental) ανάπτυξη λογισμικού. Τα κύρια χαρακτηριστικά της περιλαμβάνουν:
- **Sprints:** Χρονικά περιορισμένες περίοδοι (συνήθως 1-4 εβδομάδες) όπου ολοκληρώνεται ένα συγκεκριμένο σύνολο λειτουργιών.
- **Product Backlog:** Μια λίστα με όλες τις επιθυμητές λειτουργίες του συστήματος.
- **Ρόλοι:** 
    - *Product Owner:* Καθορίζει τις προτεραιότητες.
    - *Scrum Master:* Διασφαλίζει την τήρηση της διαδικασίας.
    - *Development Team:* Υλοποιεί το λογισμικό.
- **Τελετουργίες:** Sprint Planning (Σχεδιασμός), Daily Stand-up (Καθημερινή ενημέρωση), Sprint Review & Retrospective (Απολογισμός).

## 3. SPRINT 1

### 3.1 Καταγραφή/Συλλογή Απαιτήσεων
Στο πρώτο Sprint, επικεντρωνόμαστε στις θεμελιώδεις λειτουργίες που επιτρέπουν στον χρήστη να εισέλθει στο σύστημα και να πραγματοποιήσει μια βασική κράτηση.

#### 3.1.1 Ιστορίες Χρηστών (User Stories)
1. **Εγγραφή Χρήστη:** "Ως επισκέπτης, θέλω να μπορώ να δημιουργήσω έναν λογαριασμό παρέχοντας το email και τον κωδικό μου, ώστε να έχω πρόσβαση στις υπηρεσίες κράτησης."
2. **Αυθεντικοποίηση (Login):** "Ως εγγεγραμμένος χρήστης, θέλω να συνδέομαι με ασφάλεια στο σύστημα, ώστε να διαχειρίζομαι τις κρατήσεις μου."
3. **Προβολή Εκδηλώσεων:** "Ως χρήστης, θέλω να βλέπω μια λίστα με όλες τις διαθέσιμες εκδηλώσεις, ώστε να επιλέξω αυτή που με ενδιαφέρει."
4. **Δημιουργία Εκδήλωσης:** "Ως Διοργανωτής, θέλω να μπορώ να καταχωρώ μια νέα εκδήλωση (τίτλος, ημερομηνία, αριθμός εισιτηρίων), ώστε να ξεκινήσει η προπώληση."
5. **Κράτηση Εισιτηρίων:** "Ως Πελάτης, θέλω να δεσμεύω έναν συγκεκριμένο αριθμό εισιτηρίων για μια εκδήλωση, ώστε να προχωρήσω στην ολοκλήρωση της παραγγελίας μου."

#### 3.1.2 Sprint Planning (Προτεραιότητες)
Κατά τον σχεδιασμό του Sprint 1, οι εργασίες ταξινομήθηκαν με βάση την κρισιμότητά τους:

| Προτεραιότητα | Εργασία (Task) | Περιγραφή |
| :--- | :--- | :--- |
| **Υψηλή** | Setup Υποδομής | Ρύθμιση Docker, PostgreSQL και FastAPI boilerplate. |
| **Υψηλή** | User Management | Υλοποίηση Registration και JWT Authentication. |
| **Υψηλή** | Event CRUD | Δυνατότητα δημιουργίας και προβολής εκδηλώσεων από τη βάση. |
| **Μεσαία** | Order Logic | Βασική ροή δημιουργίας παραγγελίας και μείωσης αποθέματος εισιτηρίων. |
| **Μεσαία** | Validation | Έλεγχος διαθεσιμότητας και εγκυρότητας ημερομηνιών. |
| **Χαμηλή** | Logging | Βασική καταγραφή ενεργειών για αποσφαλμάτωση. |

---
**Τεχνική Σημείωση:** Για την υλοποίηση των παραπάνω χρησιμοποιείται το framework **FastAPI** για την ταχύτητα και την ασφάλεια τύπων (Type Safety), και η **SQLAlchemy** ως ORM για την επικοινωνία με τη βάση δεδομένων.

## 4. SPRINT 2

### 4.1 Καταγραφή/Συλλογή Απαιτήσεων
Στο δεύτερο Sprint, η εστίαση μετατοπίζεται στη διασφάλιση της ευρωστίας του συστήματος και στην αυτοματοποίηση των διαδικασιών διαχείρισης εισιτηρίων και παραγγελιών.

#### 4.1.1 Ιστορίες Χρηστών (User Stories)
1. **Αυτοματοποιημένη Ακύρωση Παραγγελιών:** "Ως διαχειριστής του συστήματος, θέλω οι παραγγελίες που δεν ολοκληρώνονται εντός 5 λεπτών να ακυρώνονται αυτόματα, ώστε να απελευθερώνονται τα δεσμευμένα εισιτήρια."
2. **Διαχείριση Κατάστασης Εκδηλώσεων:** "Ως Διοργανωτής, θέλω η κατάσταση μιας εκδήλωσης να αλλάζει (από προγραμματισμένη σε ενεργή ή ολοκληρωμένη), ώστε να ελέγχεται η ροή των πωλήσεων."
3. **Έκδοση Ψηφιακών Εισιτηρίων:** "Ως Πελάτης, θέλω μετά την επιτυχή πληρωμή να εκδίδονται τα ψηφιακά μου εισιτήρια με μοναδικούς κωδικούς."
4. **Ιστορικό Παραγγελιών:** "Ως χρήστης, θέλω να βλέπω την κατάσταση των παραγγελιών μου (Σε εξέλιξη, Ολοκληρωμένη, Ακυρωμένη, Ληγμένη)."

#### 4.1.2 Sprint Planning (Προτεραιότητες)
Κατά τον σχεδιασμό του Sprint 2, οι εργασίες ταξινομήθηκαν ως εξής:

| Προτεραιότητα | Εργασία (Task) | Περιγραφή |
| :--- | :--- | :--- |
| **Υψηλή** | Background Scheduler | Ενσωμάτωση του APScheduler για την εκτέλεση περιοδικών εργασιών στο παρασκήνιο. |
| **Υψηλή** | Order Cleanup Service | Υλοποίηση λογικής για τον εντοπισμό και την ακύρωση ληγμένων παραγγελιών (5-λεπτο timeout). |
| **Υψηλή** | Ticket Lifecycle | Δημιουργία και διαχείριση των εισιτηρίων μετά την ολοκλήρωση της παραγγελίας. |
| **Μεσαία** | Event Transitions | Αυτοματοποίηση μετάβασης καταστάσεων εκδηλώσεων (Scheduled -> Active -> Finished). |
| **Μεσαία** | Advanced Logging | Κεντρικό σύστημα καταγραφής σφαλμάτων και ενεργειών για καλύτερο monitoring. |
| **Χαμηλή** | Email Service Stub | Προετοιμασία υποδομής για μελλοντική αποστολή ενημερωτικών email. |

---
**Τεχνική Σημείωση:** Η χρήση του **APScheduler** διασφαλίζει ότι οι "ορφανές" κρατήσεις δεν θα δεσμεύουν επ' αόριστον τη χωρητικότητα μιας εκδήλωσης. Η αποδέσμευση των εισιτηρίων γίνεται με τη χρήση transactions για τη διασφάλιση της ακεραιότητας των δεδομένων.
